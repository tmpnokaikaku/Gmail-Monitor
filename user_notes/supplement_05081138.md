## 1. フロントエンドの通知先多様化
### 概要
現在の通知先は LINE を中心に設計されているが、将来的には LINE 以外のフロントエンド向け通知手段へ拡張したい。
候補としては Discord、メール再送、Web UI 上の通知、その他チャットサービスが考えられる。

### 検討理由
- LINE 固定だと利用者層を広げにくい
- マルチユーザー化した場合、ユーザーによって望む通知先が異なる
- Web アプリとして展開するなら、アプリ内通知や通知設定画面が必要になる
- Discord などは Bot / Webhook ベースで実装しやすい可能性がある

### 今後の設計論点
- 通知手段を抽象化し、LINE を1実装として扱う構成にできるか
- 通知先設定をユーザーごとに持つ方法
- 複数通知先を併用可能にするか
- 通知失敗時の再試行やフォールバック戦略をどうするか

### 現時点の方向性
- 通知処理は将来的に provider 抽象化する
- `LINE only` の前提を外せるインターフェースへ寄せる
- まずは `LINE`, `Discord` を候補として比較する

## 2. コーディングエージェントの安全な活用基盤
### 概要
今後、コード生成・調査・保守にコーディングエージェントを継続利用する前提なら、安全に使える開発基盤を整える必要がある。

### 検討理由
- 現在は `.env` や `credentials.json` に機微情報が近接しており、コードベースの走査や操作時にリスクが高い
- 将来的に AI エージェントへ実装補助や調査補助を任せるなら、アクセス可能領域を整理する必要がある
- マルチユーザー化後は、開発者以外のデータが混ざるため、より厳格な扱いが求められる

### 想定される整備項目
- シークレットの外部注入化
- 開発環境と本番環境の明確な分離
- エージェントが読むべき領域と読ませない領域の分離
- ダミーデータ / モックデータを使った安全なローカル検証環境
- 権限分離された作業用ユーザーや実行環境

### 今後の設計論点
- 本番 secrets をコードベースから完全に切り離せるか
- ローカル開発時に、安全なサンプルデータでどこまで再現できるか
- AI エージェントが扱う対象を「コード」「設定テンプレート」「ダミーDB」に限定できるか
- どの粒度で監査ログや作業記録を残すべきか

### 現時点の方向性
- Infisical などの SaaS で運用 secrets を外出しする
- 本番用 `.env` や認証ファイルをエージェント作業領域から切り離す
- 将来的には、AI 活用前提の安全な開発モードを用意する

## 3. Python から別言語への書き直し
### 概要
将来的に、Python 実装から別言語へ書き直す可能性がある。
主な関心は高速性、保守性、型安全性、並行処理のしやすさにある。

### 検討理由
- 現時点でも「処理が遅い」という不満がある
- 将来的にユーザー数や処理件数が増える場合、現行構成のままではボトルネックになりうる
- マルチユーザー化後は、ジョブ管理、非同期処理、Web API、通知配信、DB アクセスなどの責務が増える

### 候補になりうる方向
- Go
- Rust
- TypeScript / Node.js

### 各候補のざっくりした論点
- Go:
  - Web API、ジョブ実行、並行処理が比較的扱いやすい
  - バイナリ配布しやすい
  - Docker との相性がよい
- Rust:
  - 性能と安全性は高い
  - ただし実装コストや学習コストが高い
- TypeScript / Node.js:
  - フロントエンドとの親和性が高い
  - Web UI を強化する場合に統一感を出しやすい

### 注意点
- 現状の主な遅延が「言語由来」か「外部 API 待ち由来」かは未確定
- Gmail API、Gemini API、LINE/Discord API の待ち時間が支配的なら、言語変更だけでは改善が限定的かもしれない
- 先に計測と責務分離を行い、本当に書き換える価値があるか判断するべき

### 現時点の方向性
- まずは Docker 化・Secrets 分離・DB 化を優先する
- 書き直し判断は、処理計測とマルチユーザー設計の見通しが立ってから行う

## 4. Gmail OAuth / 認証周りの将来整理
### 概要
現状では `credentials.json` と `token.json` ベースの設計になっているが、マルチユーザー化にあたっては見直しが必要である。

### 検討理由
- 単一ユーザー前提の token 保持方式では拡張しにくい
- ユーザーごとの Gmail 接続状態をどう持つかが今後の主要論点になる
- OAuth の callback や再認証導線はユーザー単位で扱える必要がある

### 将来的な論点
- ユーザーごとの token を DB にどう安全に保存するか
- 再認証フローを UI 上でどう提供するか
- 失効時の再接続体験をどうするか
- ユーザー自身にどこまで Google 認証を委ねるか

## 5. LINE Webhook の位置づけ再整理
### 概要
調査結果では、LINE Developers 側設定と nginx 経路に不整合の疑いがあった。
これを単なる不具合として直すだけでなく、将来的に LINE Webhook をどこまで必要とするか整理したい。

### 検討理由
- 将来的に LINE 以外の通知手段を増やすなら、LINE 固有の入口設計をどこまで維持するか再考余地がある
- 通知専用なら push 中心でもよい可能性がある
- follow イベントや inbound メッセージを今後使うのかを整理する必要がある

### 今後の論点
- LINE は push 通知専用に割り切るか
- Webhook を今後の設定 UI や認証補助に活かすか
- 通知チャネルごとに inbound/outbound を分けるか

## 6. Web UI / API の役割拡張
### 概要
現在 nginx では `/user` と `/api/me` がアプリ側へ流れるように予約されている。
このことから、将来的に簡易 UI や API を育てる余地がある。

### 検討理由
- マルチユーザー化には、通知先設定、フィルタ管理、接続状態確認などの UI が必要になる
- OAuth 接続状態や通知履歴も UI から見られる方が望ましい
- 管理画面やユーザー画面を分離する可能性がある

### 今後の論点
- `/user` をユーザー向け画面として本格運用するか
- `/api/me` を SPA 用 API の一部として拡張するか
- `/admin` を将来の管理画面に使うか

## 7. バックエンド構成の分割
### 概要
現状は one-shot 実行と Flask 待受が同一プロセス・同一責務に近い形で混在している。
将来的には責務分割が必要になる可能性が高い。

### 想定される分割候補
- API サーバ
- ジョブ実行ワーカー
- 通知ワーカー
- OAuth / Webhook 待受

### 検討理由
- マルチユーザー化すると、リアルタイム性とバッチ性が混ざる
- 通知再試行、ユーザー操作、定期取得を分離した方が保守しやすい
- コンテナ化後はプロセス分割の境界を作りやすい

## 8. データベース導入後の設計論点
### 概要
DB 導入は既定路線に近いが、採用後の扱いはまだ固まっていない。

### 論点
- RDB を使うか
- PostgreSQL を第一候補とするか
- ユーザー設定、OAuth token、通知先設定、履歴をどこまで正規化するか
- フィルタテンプレートと抽出テンプレートの分離方法

### 現時点の方向性
- 将来的には PostgreSQL を第一候補として考える
- ただし、初期段階では構造を決めすぎず、必要最小限のテーブルから始める方がよい
