# プロジェクト改造方針
作成日時は日本時間基準。

## 目的
このプロジェクトを、現在の個人利用・単一ユーザー前提の Gmail 監視アプリから、将来的な拡張に耐えられる構成へ段階的に改造する。

今回の改造方針で特に重視する点は次の3つ。
- Docker化
- シークレット管理SaaSの導入
- マルチユーザー化に向けた拡張性確保

## 現状認識
現在のシステムは、OCI 上で `systemd timer` により定刻起動される one-shot 型の Python アプリである。
公開経路は `Cloudflare -> nginx -> Flask(127.0.0.1:8080)` であり、アプリ本体は `app.py` を直接実行している。

また、設定と機微情報は `.env`、`credentials.json`、`token.json` としてアプリディレクトリ直下に置かれている。
この構成は個人運用としては成立しているが、以下の問題を抱える。

- シークレット管理がローカルファイル直置きである
- Docker 化されておらず、環境再現性が弱い
- 単一ユーザー前提で、マルチユーザー化に向かない
- ユーザー設定と運用設定が分離されていない
- 一部の Webhook / 入口設定に不整合の疑いがある

## 改造の基本方針
一気に全面刷新するのではなく、以下の順序で段階的に進める。

1. 既存運用を壊さない範囲で Docker 化する
2. `.env` 直置きをやめ、Secrets Manager 的な仕組みに移行する
3. ユーザー固有データをファイルから分離し、DB 前提の構成へ寄せる
4. そのうえでマルチユーザー対応に進む

この順序を取る理由は、入口のネットワーク設定、OAuth、Webhook、既存通知フローを先に壊さないためである。

## 1. Docker化方針
### 基本方針
Docker 化の第一段階では、`app` だけをコンテナ化する。
Cloudflare、nginx、現行ドメイン、systemd timer は当面維持する。

つまり、現在 `gmm.service` が直接起動している Python 実行部分だけを Docker コンテナに差し替える。

### この方針の理由
- 現在の公開経路を大きく変えずに済む
- OAuth や Webhook の設定影響を最小化できる
- 問題が起きたときに、Python 直起動へ比較的容易に切り戻せる
- Docker 導入による効果を最小変更で得られる

### 第一段階で維持するもの
- Cloudflare
- nginx
- 現行ドメイン
- systemd timer

### 第一段階で置き換えるもの
- `gmm.service` の `ExecStart`
- OCI 上の Python 仮想環境依存

### 第一段階の実装目標
- Dockerfile を作成する
- 実環境ベースで依存関係を整理する
- `127.0.0.1:8080` で現在と同じ待受ができるようにする
- timer 起動時にコンテナ実行できるようにする

### Docker 化の前提条件
Docker 化の前に、次の点を確認・解消する必要がある。

- nginx と LINE Webhook `/callback` の経路不整合の解消
- `requirements.txt` と OCI 実環境依存との差分整理
- 外部注入対象ファイルの整理

## 2. シークレット管理SaaS導入方針
### 基本方針
OCI ネイティブな Secrets Manager には寄せず、Secrets 管理は SaaS を採用する。
現時点では Infisical を第一候補とする。

### 採用理由
- OCI 依存を強めたくない
- 将来、OCI 以外へ移る可能性がある
- ローカル・CI・本番の共通運用に寄せやすい
- 小規模な無料運用から始めやすい

### SaaS 導入の役割
Infisical には、アプリ全体の運用用シークレットのみを置く。
外部ユーザーごとのデータや設定を直接 Infisical に載せる設計は取らない。

### Infisical に置く想定のもの
- `LINE_CHANNEL_ACCESS_TOKEN`
- `LINE_CHANNEL_SECRET`
- `GEMINI_API_KEY`
- `SERVER_DOMAIN`
- Google OAuth client 情報 (`credentials.json` 相当)
- 内部用の署名鍵や管理者向け secret
- ログ設定や運用設定のうち外部注入したいもの

### Infisical に置かない想定のもの
- 外部ユーザーのプロフィール
- 外部ユーザーの通知先情報
- 外部ユーザーの Gmail OAuth token
- 外部ユーザーごとのフィルタ設定
- ユーザーごとの処理履歴

### 管理単位の方針
Infisical は次のような位置づけで使う。

- admin identity: 管理者用
- machine identity: アプリ実行用

外部ユーザーごとに identity を切る方針は取らない。
これにより、無料枠で運用しやすくしつつ、ユーザー数増加によって Infisical の identities 制限へ早期に達することを避ける。

## 3. マルチユーザー化に向けた拡張方針
### 基本方針
マルチユーザー化では「運用 secrets」と「ユーザー固有データ」を明確に分離する。

- 運用 secrets: Infisical
- ユーザー固有データ: アプリケーション DB

この責務分離を早い段階で導入することが、将来の拡張性に直結する。

### まず解消すべき単一ユーザー前提
現状コードには単一ユーザー前提が残っている。

- `MY_LINE_UID` 固定
- `filters.json` が単一ファイル
- `token.json` が単一ユーザー分のみ
- Gmail 認証状態が単一ユーザー向け

これらは将来的に次のように置き換える必要がある。

- `MY_LINE_UID` -> `users` テーブルの通知先情報
- `filters.json` -> DB 上のユーザー別設定
- `token.json` -> ユーザー別 OAuth token レコード
- 単一 Gmail セッション -> ユーザー別接続状態管理

### マルチユーザー化のためのデータ分離
将来的には、少なくとも次のデータ領域を分ける。

- ユーザー基本情報
- 通知先設定
- Gmail OAuth 接続情報
- フィルタ設定
- 抽出ルール適用対象
- 配信履歴

### `filters.json` の扱い
`filters.json` は現状ではユーザー設定とアプリロジックが混在している。

現在の1ルールは概ね次の3要素を持つ。
- キーワード
- 送信者
- 抽出関数

このうち、
- キーワード
- 送信者

はユーザー設定として DB に寄せやすい。

一方で、
- 抽出関数

はアプリケーション側ロジックなので、当面はコード側テンプレートとして管理し、DB 側では「どの抽出テンプレートを使うか」を参照する形が現実的である。

### マルチユーザー化に向けた当面の設計
完全な汎用抽出をいきなり目指さず、次の段階を踏む。

1. 既存の抽出テンプレートをアプリ側に残す
2. ユーザーが利用するテンプレート種別を選べるようにする
3. キーワード・送信者などの選別条件だけをユーザー設定化する
4. 将来的に LLM 利用範囲や抽出戦略を再検討する

## 4. 構成の将来像
最終的には、おおよそ次の構成を目指す。

- Cloudflare
- nginx
- app container
- DB
- Infisical

役割分担:
- Cloudflare: 外部入口
- nginx: TLS 終端と経路制御
- app container: アプリ本体
- DB: ユーザー固有データ
- Infisical: 運用シークレット

## 5. 優先順位
### 優先度 高
- nginx と `/callback` の不整合確認・解消
- Docker 化に必要な依存関係整理
- Infisical 導入前提で、`.env` 内の項目を分類する
- 不要な旧シークレットや未使用情報を削除する

### 優先度 中
- `credentials.json` / `token.json` の管理方法を再設計する
- `filters.json` をファイル依存から切り離す準備をする
- ユーザー情報を格納する DB 設計の草案を作る

### 優先度 低
- UI/API の整理
- 管理画面の強化
- 抽出テンプレートの拡張
- 本格的なマルチテナント設計

## 6. 当面の実装ロードマップ
### フェーズ1: 現行整理
- 既存の設定値を棚卸しする
- `.env` 項目を「存続」「廃止」「DB移行候補」に分類する
- `/callback` 経路不整合を事実ベースで解消する

### フェーズ2: Docker 導入
- Dockerfile 作成
- 実環境依存の反映
- `gmm.service` の起動対象を Docker 化
- OCI 上で現行運用を壊さず置換

### フェーズ3: Secrets 管理移行
- Infisical プロジェクト作成
- admin / machine identity 設計
- `.env` 依存を外部注入へ順次置換
- ローカルと本番の注入方式を統一方向へ寄せる

### フェーズ4: マルチユーザー化準備
- DB 導入
- ユーザー固有設定の保存先移行
- Gmail token のユーザー別管理設計
- 通知先の複数ユーザー対応

## 7. 結論
このプロジェクトの改造方針は、次の一文に要約できる。

「まずは既存の OCI 運用を壊さずに app を Docker 化し、Secrets 管理は Infisical のような SaaS に寄せ、ユーザー固有データは DB に分離することで、将来的なマルチユーザー化に耐える構成へ段階的に移行する。」

全面的な作り直しではなく、責務分離を先に進めることが重要である。
