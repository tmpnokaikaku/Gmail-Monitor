今後の改善方針メモ。適宜追記。

## ノートのエンコーディング
- このファイルは `UTF-8 (BOMなし)` / 改行コード `LF`。
- PowerShell の表示では文字化けして見えることがあるが、ファイル自体のエンコーディングは壊れていない。

## 開発環境の方針
- Dockerの導入：
  - ローカルで開発しリモート(OCI)にデプロイする過程の簡略化。
  - ローカルでテストをできるようにしたい。
  - 「ローカル」もWindows環境とWSL環境をそれぞれ使えるようにしたい。
  - OCIから別の環境に引っ越す可能性も念頭に。
  - 開発経験/勉強のため。
- 安全なシークレット管理方法の導入：
  - 外部のシークレット管理サービスなどを用いて、機微情報の管理を厳格化。
  - AIエージェントがプロジェクト全体を走査できるように、シークレットは外部や隔離区間から注入するのが望ましい。
  - 現在の自分専用アプリから、将来的なリリースに向けて必要に思われる。

## 実装済み機能に対する不満と調査改善の方針
- Gmailを使用するため、Googleアカウントの認証が必要である。`credentials.json`と`token.json`を用いて手動ログインに替えているが、定期的に手動再認証が求められる。
- プログラムの実行がとても遅い。

## アプリ機能の拡張性確保の方針
- マルチユーザー対応：
  - `filters.json`のようなコンフィグもユーザーごとに持てるようにする。将来的にはデータベース化も検討。
  - 通知先や OAuth 情報、フィルタ条件をユーザー単位で分離する必要がある。
- Webアクセスの改善：
  - 現在は `systemd` で定期時刻に発火して、OCI内でFlaskサーバーを立ち上げており、そのFlaskサーバーが nginx によって外部に公開されている。なので、サーバーが立っていないときブラウザからURLを直接叩いてもアクセスできない。
  - コンフィグ設定用のエンドポイントを作成してある(Cloudflareを導入済)ので、ここから常時コンフィグ設定できるようにしたい。
- 転送先の多様化：
  - 現在LINEだけだが、Discordや他のアプリとも連携できるようにしたい。
  - マルチユーザー化するにあたって、LINEは少し大変そうかも？
- メールフィルター条件の改善：
  - 対象となるドメインや要件の選別の改善。
- 将来的な追加機能の導入アイデア：
  - メールに対して、内容や期限をもとに重要度をスコアリングする。
  - シラバスや履修登録ポートフォリオの活用。

## 既知の要件
- Google OAuthは、有効なドメインつきURLを要求する。IPアドレス直の `https://xxx.xxx.xx.xx` のようなものは無効。
- 現状コードは OCI 側のネットワーク設定、環境変数、認証ファイル、仮想環境を前提としているため、ローカルでそのまま動かすことは前提にしない。
- Docker 導入時も、いきなり本番構成全体を置き換えるのではなく、まずはアプリ本体だけをコンテナ化し、`nginx`・DNS・証明書・Cloudflare は当面維持する方が安全。

## 現在の運用整理
- 現在のアプリは常駐監視ではなく、基本的には「起動 -> Gmail取得 -> フィルタ判定 -> 抽出 -> LINE通知 -> 終了」に近い構成。
- Flask サーバーは OAuth コールバックや LINE Webhook を受けるために必要だが、常時起動されているわけではない可能性がある。
- `systemd service` で常駐させているのか、`systemd timer` や cron 的に定期起動しているのかは Docker 設計に影響するため、要確認。

## 永続化・機密情報の整理
- Docker 化してもコンテナイメージに含めない方がよいファイル：
  - `credentials.json`
  - `token.json`
  - `.env` 相当の環境変数定義
  - ログ出力先
- 特に `token.json` は OAuth 後に更新されるため、bind mount か volume 前提で扱う必要がある。
- 現在の `credentials.json` と `token.json` はリポジトリ配下に存在しているため、運用上は改善余地が大きい。

## 公開経路と責務分担
- 現状の公開経路はおそらく `Cloudflare -> nginx -> Flask`。
- どこで TLS 終端しているか、`nginx` が `proxy_pass` している先はどこか、Cloudflare をどこまで使っているかを明確にする必要がある。
- Docker 導入時に `nginx` までコンテナ化するのか、ホスト側に残すのかを先に決めるべき。

## アプリ側の既知課題
- `filters.json` に `sdsOffice` がある一方で、現行コードの抽出実装は `manaba` と `cels` のみ。`sdsOffice` に一致したメールは現状通知されない。
- コードやログ文言の一部に文字化けが見られる。実行時のログ可読性や、本文抽出ルールへの影響有無を確認したい。
- テストが不足している。Gmail取得、フィルタ判定、抽出、通知の最低限の自動テストを後で追加したい。

## 調査と検討
- コード外の環境、導入しているサービスやフレームワークの調査：
  - systemd
  - nginx
  - Cloudflare
  - Google OAuth
  - venv
  - 他(あれば)
- OCI 側で確認すべき事項：
  - `systemd` の unit 内容
  - 起動コマンド
  - `nginx` の `server_name` と `proxy_pass`
  - 環境変数の配置場所
  - 証明書の管理場所
  - ログの確認手段
- Docker 化の第一段階：
  - まずはアプリ本体だけコンテナ化する。
  - 既存の `nginx` と DNS 設定はできるだけ維持する。
  - ローカルではモックやダミー設定を使って、Gmail/LINE/Cloudflare なしでも最低限の起動確認をできる状態を目指す。
- 実行速度とマルチユーザー化を念頭に置いた、別言語へのリファクタリングは中長期課題。
- OCIでAIエージェント用のユーザーを作成する案は、権限分離と監査性を確保できるなら検討余地あり。
