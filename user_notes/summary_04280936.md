## 1. 現行システム構成の要約
現行の Gmail Monitor は、OCI 上で `systemd timer` により定刻起動される Python アプリケーションである。
常駐監視ではなく、概ね「起動 -> Flask 待受開始 -> Gmail取得 -> フィルタ判定 -> 情報抽出 -> LINE通知 -> 終了」という one-shot 的な構成になっている。

公開経路は次の通り。

`Cloudflare -> nginx -> Flask(127.0.0.1:8080)`

アプリ本体は以下の前提で動作している。
- 実行ユーザー: `opc`
- 作業ディレクトリ: `/opt/gmm_project/Gmail-Monitor`
- 実行コマンド: `/opt/gmm_project/GmmEnv/bin/python -u app.py`
- 環境変数注入: `/opt/gmm_project/Gmail-Monitor/.env`
- OAuth / トークン関連ファイル: `credentials.json`, `token.json`
- ログ出力先: `log/gmm_app.log` および `journalctl`

## 2. systemd / 実行方式
`gmm.service` は手動常駐ではなく、timer から起動される one-shot 的な service である。

### 平日タイマー
- 08:25
- 10:20
- 12:50
- 14:50
- 16:50
- 18:45
- 21:20

### 週末タイマー
- 07:00
- 12:00
- 17:00
- 22:00

両 timer とも `Persistent=true` になっている。
そのため、停止中の取りこぼし時刻の扱いにも systemd の設定が影響する。

## 3. nginx / Cloudflare / 公開経路
現行ドメインは `NAME.dpdns.org`。
Cloudflare 側ではこの現行ドメインのみ登録されていた。

### Cloudflare
- SSL mode: `Full (strict)`
- Automatic mode: 無効
- Security rules: 特別なルールなし
- Edge Certificates: Cloudflare 管理の証明書が有効

### nginx
nginx は Cloudflare Origin CA を用いて TLS を終端している。

80番ポート:
- 現行ドメインへのアクセスを HTTPS に 301 リダイレクト

443番ポート:
- 必要パスのみ Flask へ proxy
- それ以外は原則隠蔽

proxy 対象:
- `/oauth/`
- `/user`
- `/api/me`

非公開/隠蔽:
- `/` は 404
- `/admin` は 404

また、Flask が落ちているときのフォールバックも nginx 側で定義されている。
- `/oauth/` は案内 HTML を 200 で返す
- `/user` は 503 HTML
- `/api/me` は 503 JSON

## 4. OAuth / LINE Webhook の状態
### Google OAuth
- Google Cloud Console には現行ドメインの `/oauth/callback` が登録済み
- 旧 `duckdns` ドメインも残っている
- `credentials.json` の中身には古い `ngrok` 時代の URI が残っている
- JavaScript origins は未登録

これより、実運用上の OAuth は Google Cloud Console 側の設定で成立しており、ローカルファイル中の古い URI は設定残骸とみられる。

### LINE Webhook
- LINE Developers Console には現行ドメインの `/callback` が登録済み
- Webhook 利用は有効
- Webhook 再送は無効
- Verify 結果は 404

### 重要な不整合
nginx 設定を確認した限り、`/callback` を Flask に流す `location` が存在しない。
実際の疎通確認でも、`https://NAME.dpdns.org/callback` は 404 である。

一方で、`https://NAME.dpdns.org/oauth/callback` は 200 を返しており、OAuth の待受ページ側は nginx 設定と整合している。

このため、少なくとも現時点の設定確認結果だけを見ると、LINE Webhook 設定と nginx の公開設定には不整合がある。

## 5. DNS 状態
外部から見た `NAME.dpdns.org` の DNS 応答は Cloudflare の IP を返している。

確認できたこと:
- A レコードあり
- AAAA レコードあり
- 応答先は Cloudflare

したがって、ユーザーや外部サービスは OCI の生 IP ではなく Cloudflare 経由で現行ドメインに到達している。

## 6. Python / 依存関係
OCI 側の Python は 3.9.21。
ローカル開発側は 3.13.3。

OCI 実環境では `GmmEnv` 仮想環境に、`requirements.txt` より多くの依存パッケージが導入されている。
そのため、現行の `requirements.txt` だけでは OCI 環境を完全再現できない可能性がある。

Docker 化する場合、依存関係は OCI 実環境に寄せて再定義する必要がある。

## 7. ファイル配置と永続化対象
現行プロジェクトディレクトリには、コードだけでなく設定や機微情報が同居している。

主要ファイル:
- `.env`
- `credentials.json`
- `token.json`
- `filters.json`
- `log/`

Docker 化観点では、少なくとも以下は永続化または外部注入対象になる。
- `.env`
- `credentials.json`
- `token.json`
- `filters.json`
- `log/`

特に `token.json` は実行中に更新されるため、読み取り専用では扱えない。

## 8. セキュリティと設計上の懸念
### 1. 機微情報の保管
`.env`、`credentials.json`、`token.json` に機微情報が含まれている。
これらがアプリケーションディレクトリ直下に置かれている現在の設計は、将来的な AI 利用やマルチユーザー化を考えると望ましくない。

将来的には、Secrets Manager 相当の外部注入方式へ移行することが望ましい。

### 2. 権限設定
OCI 上のプロジェクト配下には権限がかなり緩いファイルがある。
少なくとも機密ファイルについては権限を厳格化した方がよい。

### 3. filters.json の将来性
`filters.json` は現状ユーザー固有コンフィグとして使われている。
ただし、マルチユーザー化やユーザー編集機能を考えると、将来的には JSON ファイルではなく DB 等へ移行する可能性が高い。

### 4. 不要情報の残存
`.env` 内に未使用と思われる学籍情報や旧 ngrok ドメインなどが残っている。
不要情報は削除候補として整理すべき。

## 9. Docker 化に向けた結論
### 結論
Docker 導入の第一段階として最も安全なのは、`app` だけをコンテナ化し、周辺の外部構成は当面維持する方針である。

維持するもの:
- Cloudflare
- nginx
- 現行ドメイン
- systemd timer

置き換えるもの:
- `gmm.service` が直接起動している Python 実行部分

つまり、`127.0.0.1:8080` の実体を「OCI 上の Python 直起動」から「Docker コンテナ内アプリ」に差し替える方針が最小リスクである。

### この方針が妥当な理由
- 現在の公開経路と OAuth 設定を壊しにくい
- Cloudflare / nginx / ドメイン設定をそのまま活かせる
- timer もそのまま活かせる
- 問題発生時の切り戻しが比較的簡単

## 10. Docker 化前に片付けるべき事項
優先度順に並べる。

### 優先度 高
- nginx と LINE Webhook `/callback` の不整合を解消する
- `requirements.txt` を実環境ベースで見直す
- `.env` と `credentials.json` と `token.json` の扱い方針を決める

### 優先度 中
- 不要な旧 `duckdns` / `ngrok` 関連設定の整理
- 旧学籍情報など未使用機微情報の削除
- 機密ファイルの権限厳格化

### 優先度 低
- `filters.json` の将来 DB 化検討
- マルチユーザー化に向けた設計変更

## 11. 現時点での最重要論点
最も重要なのは、LINE Developers Console では `/callback` が設定されているのに、nginx 側では `/callback` を公開していないように見える点である。

この点を曖昧なまま Docker 化に進むと、移行後に Webhook だけ壊れるリスクがある。
したがって、Docker 化の前にまずこの経路不整合を事実ベースで解消するべきである。

## 12. 一言でまとめると
現行構成は「Cloudflare と nginx に支えられた、timer 起動の one-shot Flask アプリ」である。
Docker 化は十分可能だが、最初から全面コンテナ化するのではなく、まずはアプリ本体だけをコンテナ化し、入口設定とシークレット管理を先に整えるのが現実的である。
