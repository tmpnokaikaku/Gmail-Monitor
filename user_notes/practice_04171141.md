# OCI調査手順メモ
Docker導入や今後の運用改善のために、OCI上で自分が確認するべきことを整理する。
このメモは「何を確認するか」「なぜ必要か」「どう調べるか」をまとめたもの。

## 使い方
- まず SSH で OCI に接続する。
- 以下の項目を上から順に確認する。
- コマンド結果は必要な部分だけこのメモに転記するか、別メモに貼る。
- 認証情報や秘密鍵そのものは貼らない。

## 1. 現在のアプリの起動方式を確認する
### 目的
- このアプリが常駐型なのか、定期実行型なのかを確定する。
- Docker 化するときに `docker run` 常駐にするのか、cron/timer 的に動かすのかを判断する。

### 確認すること
- `systemd service` で常駐しているのか
- `systemd timer` で定期実行しているのか
- あるいは cron を使っているのか
- 実際の起動コマンドは何か

### 手順
```bash
systemctl list-units --type=service | grep -i gmm
systemctl list-units --type=timer | grep -i gmm
systemctl status gmm.service
systemctl cat gmm.service
systemctl list-timers --all | grep -i gmm
crontab -l
sudo crontab -l
```

### 見るポイント
- `ExecStart=` に何が書かれているか
- `WorkingDirectory=` はどこか
- `Environment=` や `EnvironmentFile=` を使っているか
- `Restart=` の設定は何か
- timer があるなら、何分ごと・何時ごとか

## 2. プロジェクト配置場所と実行ユーザーを確認する
### 目的
- どのディレクトリを基準に動いているかを確定する。
- Docker 化時の bind mount や volume 設計の材料にする。

### 確認すること
- 実際のプロジェクト配置ディレクトリ
- 実行ユーザー名
- 該当ファイルの所有者と権限

### 手順
```bash
pwd
whoami
systemctl cat gmm.service
ls -la
ls -la /path/to/project
```

### 見るポイント
- `WorkingDirectory` と実ファイル配置が一致しているか
- 実行ユーザーに `token.json` やログファイルの書き込み権限があるか

## 3. Python 仮想環境と依存関係を確認する
### 目的
- 現在どの Python で動かしているかを把握する。
- Dockerfile を書くときのベースイメージや依存導入方法の参考にする。

### 確認すること
- Python の実体パス
- 仮想環境の場所
- インストール済みパッケージ

### 手順
```bash
which python
which python3
python --version
python3 --version
systemctl cat gmm.service
source /path/to/venv/bin/activate
pip list
pip freeze
```

### 見るポイント
- `ExecStart` が venv 内の Python を使っているか
- `requirements.txt` と実際の `pip freeze` に差がないか

## 4. 環境変数と設定ファイルの置き場所を確認する
### 目的
- Docker 化後にコンテナへどう設定注入するかを整理する。
- 秘密情報をイメージに含めない方針を具体化する。

### 確認すること
- `.env` を使っているか
- `EnvironmentFile` を使っているか
- 必須環境変数は何か
- `credentials.json` と `token.json` はどこに置いているか

### 手順
```bash
systemctl cat gmm.service
ls -la /path/to/project
ls -la /path/to/project/.env
ls -la /path/to/project/credentials.json
ls -la /path/to/project/token.json
grep -R "LINE_CHANNEL_ACCESS_TOKEN" /path/to/project
grep -R "SERVER_DOMAIN" /path/to/project
grep -R "MY_LINE_UID" /path/to/project
grep -R "GEMINI_API_KEY" /path/to/project
```

### 見るポイント
- `.env` がどこにあるか
- systemd が `.env` を読む仕組みか、シェルで export 済みか
- `token.json` が更新される場所に書き込みできるか

## 5. nginx の公開設定を確認する
### 目的
- Flask アプリがどう外部公開されているかを把握する。
- Docker 化後も `nginx -> app container` の形を維持できるか判断する。

### 確認すること
- `server_name`
- `listen`
- `location`
- `proxy_pass`
- SSL 証明書の設定

### 手順
```bash
sudo nginx -t
sudo nginx -T | less
sudo grep -R "server_name" /etc/nginx
sudo grep -R "proxy_pass" /etc/nginx
sudo grep -R "ssl_certificate" /etc/nginx
sudo grep -R "ssl_certificate_key" /etc/nginx
```

### 見るポイント
- 受けているドメイン名は何か
- どのパスをどこへ転送しているか
- `proxy_pass http://127.0.0.1:8080` のようになっているか
- OAuth コールバック用の `/oauth/*` と LINE の `/callback` が正しく通る設定か

## 6. ドメインと Cloudflare の関係を確認する
### 目的
- ドメインが OCI のどこに向いているかを把握する。
- OAuth や Webhook の入口を壊さないために必要。

### 確認すること
- 利用中のドメイン名
- Cloudflare をプロキシとして使っているか
- DNS レコードが OCI の IP を向いているか

### 手順
- これは OCI サーバー上だけでは完結しない可能性がある。
- まず nginx 設定や `.env` からドメイン名を確認する。
```bash
systemctl cat gmm.service
sudo nginx -T | grep server_name
grep -R "SERVER_DOMAIN" /path/to/project
curl -I https://your-domain.example.com
dig your-domain.example.com
nslookup your-domain.example.com
```

### 見るポイント
- `SERVER_DOMAIN` の値
- Cloudflare のオレンジクラウド運用か DNS only か
- OAuth 設定に登録されているドメインと一致しているか

## 7. Google OAuth 設定との整合性を確認する
### 目的
- OAuth フローが今なぜ通っているのか、何を維持すべきかを明確にする。
- Docker 化や移設時に壊れやすい箇所を把握する。

### 確認すること
- `credentials.json` の redirect URI
- 実際の公開 URL
- `SERVER_DOMAIN` と一致しているか

### 手順
```bash
cat /path/to/project/credentials.json
grep -R "redirect_uri" /path/to/project
grep -R "SERVER_DOMAIN" /path/to/project
```

### 見るポイント
- OAuth クライアント設定で許可された URI と、実際に使っている `https://<domain>/oauth/callback` が一致するか
- IP アドレス直打ちではなく、ドメイン名ベースで運用されているか

## 8. LINE Webhook / 通知設定を確認する
### 目的
- LINE 通知がどう成立しているかを確認する。
- マルチユーザー化や転送先追加の制約を把握する。

### 確認すること
- `LINE_CHANNEL_ACCESS_TOKEN`
- `LINE_CHANNEL_SECRET`
- `MY_LINE_UID`
- Webhook の受け口 URL

### 手順
```bash
systemctl cat gmm.service
grep -R "LINE_CHANNEL_ACCESS_TOKEN" /path/to/project
grep -R "LINE_CHANNEL_SECRET" /path/to/project
grep -R "MY_LINE_UID" /path/to/project
grep -R "/callback" /path/to/project
```

### 見るポイント
- 自分専用の `MY_LINE_UID` 固定で動いているか
- 将来のマルチユーザー化でどこを変える必要があるか

## 9. ログの見方と障害時の確認方法を確定する
### 目的
- 問題発生時の確認手段を整理する。
- Docker 化の前後で、最低限どこを見ればよいか明確にする。

### 確認すること
- アプリログの出力先
- `journalctl` で見えるか
- nginx ログの場所

### 手順
```bash
journalctl -u gmm.service -n 100 --no-pager
journalctl -u gmm.service -f
sudo tail -n 100 /var/log/nginx/access.log
sudo tail -n 100 /var/log/nginx/error.log
systemctl cat gmm.service
```

### 見るポイント
- アプリの標準出力が journald に流れているか
- Flask 側のエラーと nginx 側のエラーを切り分けられるか

## 10. Docker 化のために先に決めるべきこと
### 目的
- 調査結果を設計判断につなげる。

### 決めること
- アプリだけ先にコンテナ化するか
- nginx も後でコンテナ化するか
- `token.json` と `.env` をどの方式で注入するか
- systemd からコンテナを起動するか、timer と組み合わせるか

## 調査時の注意
- `token.json` や `.env` の中身そのものは外部に共有しない。
- `credentials.json` も client secret を含むため、全文は貼らない。
- 必要なのは「どこにあるか」「どう読まれているか」「どんな値の種類か」であって、秘密値そのものではない。
